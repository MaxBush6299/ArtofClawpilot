- 2026-04-27 issue #17 auth recommendation: Tolkien analyzed authentication options for the planned API route to request manual image generation. Current architecture uses Azure Static Web Apps for frontend, Azure Container Apps Job for hosted runner, user-assigned managed identity for job IAM, Key Vault for secrets, and GitHub App tokens for repo access. **Recommended first implementation:** Static Web Apps built-in Easy Auth with Microsoft Entra ID, configured on the SWA resource via Azure portal or Bicep. Easy Auth provides zero-code authentication at the Azure platform layer before API routes execute. API route validates `.auth/me` endpoint for user identity and enforces known-user allowlist (e.g., owner email or specific AAD object IDs) in route code. Rate limiting via in-memory request tracking or SWA config (if supported). Job invocation uses existing managed identity's ACA Job start RBAC permission. **Azure/IAM implications:** Grant job managed identity `Microsoft.App/jobs/start/action` on the Container Apps Job resource; no new app registration or client secret needed; SWA system-assigned MI not required (job invokes itself, not SWA). **Alternatives considered and deferred:** API Management with subscription keys (adds cost, complexity, separate APIM resource), Azure Function App with managed identity (splits API boundary from SWA, requires new Function resource and deployment wiring), custom bearer token with Key Vault secret (reinvents auth, higher maintenance). **Next steps for implementation:** Configure Easy Auth on SWA with Microsoft Entra ID provider, add role assignment for job MI to start jobs, implement API route with allowlist check, add observability for auth failures, smoke test on disposable branch before production. Decision recorded in `.squad/decisions/inbox/tolkien-issue-17-auth-options.md`.
