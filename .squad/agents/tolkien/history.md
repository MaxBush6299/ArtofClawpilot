Job invocation uses existing managed identity's ACA Job start RBAC permission. **Azure/IAM implications:** Grant job managed identity `Microsoft.App/jobs/start/action` on the Container Apps Job resource; no new app registration or client secret needed; SWA system-assigned MI not required (job invokes itself, not SWA). **Alternatives considered and deferred:** API Management with subscription keys (adds cost, complexity, separate APIM resource), Azure Function App with managed identity (splits API boundary from SWA, requires new Function resource and deployment wiring), custom bearer token with Key Vault secret (reinvents auth, higher maintenance). **Next steps for implementation:** Configure Easy Auth on SWA with Microsoft Entra ID provider, add role assignment for job MI to start jobs, implement API route with allowlist check, add observability for auth failures, smoke test on disposable branch before production. ## Learnings

- **2026-04-27 internal clawpilot agent auth recommendation:** Max asked about auth for an internal agent ("clawpilot") calling the issue #17 manual generation API. Context: API uses API key Phase 1 MVP; clawpilot is internal agent, not public frontend. **Recommended approach:** Use dedicated API key stored in Key Vault for clawpilot agent (separate from user-facing keys), enforce no frontend exposure (agent runs server-side only, never in browser), future hardening path includes managed identity/workload identity when agent runs in Azure compute (Container Apps, Function, or separate runner job). **Comparison:** (1) API key in KV: simplest first step, aligns with Phase 1 API design, per-agent keys enable revocation without service disruption; (2) Managed identity/Entra workload identity: preferred for agent-as-Azure-resource scenarios (e.g., scheduled agent job, Function-hosted agent), eliminates secret rotation, requires Azure compute boundary; (3) Private network restrictions: consider VNET integration if agent runs in ACA/Functions and API moves to same environment, deferred until multi-resource deployment; (4) No frontend exposure: enforce at deployment architecture level—agent never receives API key in client bundle, never calls from browser context. **First implementation:** clawpilot uses dedicated Key Vault API key (`clawpilot-api-key` secret), runs as server-side process (local dev CLI or Azure compute), calls API with `X-API-Key` header. **Future hardening (Phase 2/3):** when clawpilot deployed as ACA Job or Function, grant user-assigned managed identity, switch API to validate managed identity tokens instead of API key, retire secret-based auth. Decision captured in `.squad/decisions/inbox/tolkien-clawpilot-agent-auth.md`.
