- **2026-04-27 multi-run testing gates:** Max requested gate definition for API-triggered manual runs supporting multiple images per day. Current architecture enforces one-outcome-per-`runDate` (exactly one image OR one skip), with idempotent rerun no-op when same `runDate` already resolved. Proposed change: support multiple images per calendar day via composite identity (`runDate`, `requestId`), while preserving skip-closes-day behavior and duplicate request rejection. Butters created comprehensive testing gate matrix in `workspace/multi-run-testing-gates.md` covering: (1) duplicate request rejection (same `requestId` → idempotent no-op), (2) different request same day (distinct `requestId` → new image), (3) scheduled + manual coexistence, (4) skip-closes-day enforcement (no retry after skip), (5) concurrent API call safety, (6) git push/rebase retry handling, (7) frontend tolerance for multi-`runDate` entries, (8) log/call-count correlation per `requestId`. Gate checklist requires 10 acceptance tests (MR-1 through MR-10), extended hosted smoke proof (Phase B/C with multi-run scenarios), concurrent execution proof, Wendy frontend sign-off, and Log Analytics `requestId` correlation before API endpoint ships. Rejection criteria include: duplicate (`runDate`, `requestId`) in gallery, skip-then-retry creating image, concurrent corruption, missing `requestId` in logs, or false collision between scheduled and manual runs. Architecture risk areas: gallery JSON concurrency (mitigation: git pull-rebase-push with retry), asset filename collision (extend pattern to include `requestId` suffix), frontend assumptions about `runDate` uniqueness, skip retry bypass prevention. Recommended rollout: Phase 1 (manual-only multi-run) for 30+ days before Phase 2 (multi-scheduled-runs). Open questions for Max: `requestId` auto-generation vs. API-provided, asset naming strategy, skip retry escape hatch policy, scheduler multi-run scope, per-day rate limiting.
