- 2026-04-27 multi-run idempotency design: Max asked about supporting manual API-triggered runs in addition to scheduled daily runs without duplicates. Kyle designed backend contract introducing `runId` as primary idempotency key (format `{runDate}-{source}-{seq}`), keeping `runDate` as day-level granularity. Key decisions: each run gets unique `runId`, each image gets unique `imageId` with collision suffix, skip records indexed by `runId`, file naming supports multiple images per day, API route queues new runs with distinct `runId`, same `runId` rerun = no-op, git commits remain atomic one-per-publish. Three-phase migration: (1) add optional `runId` support backward-compatible, (2) implement API route with auth, (3) enforce uniqueness. Proposal written to `.squad/decisions/inbox/kyle-multi-run-idempotency.md` covering idempotency keys, data shape, file naming, CLI/API inputs, logs, retry behavior, and coexistence of scheduled + manual runs.
- 2026-04-27 API rollout lift sizing: Max asked about implementation lift for deploying multi-run support. Kyle sized two options: (1) orchestrator-only update (2-3 days: add `runId` contract, update idempotency checks, validate multi-run path, no infra changes), (2) full API integration (1-2 weeks: Option 1 plus Azure Function App, auth layer, request validation, ACA Job invocation, execution tracking, rate limiting). Key architectural decisions: direct ACA Job invocation (not enqueue-first) because job has built-in retry and queue adds little value; lock-free concurrency (orchestrator resolves via `runId` rather than advisory lock); API checks gallery state before invocation to prevent wasted job executions. Concrete touchpoints identified: `orchestrator/contracts.py` + `validation.py` + `main.py` for Option 1; new `api/trigger-run/*` Functions code + Bicep infra modules for Option 2. Recommended sequence: prove Option 1 (orchestrator contract) with manual CLI triggers first, then add API layer behind feature flag once validation gates pass. Cost estimate: ~$0.04/month incremental for API layer (negligible). Sizing doc written to `workspace/kyle-api-rollout-lift-sizing.md` covering code areas, infrastructure changes, duplicate-trigger prevention, monitoring, and three-phase rollout timeline.
