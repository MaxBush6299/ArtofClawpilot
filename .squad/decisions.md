- Multi-run invariant architecture shift: Max requested API-triggered manual runs to enable multiple images per day beyond the scheduled daily run. Current architecture enforces exactly one image OR one skip per `runDate`. Proposed shift: introduce `runId` (unique run identifier) as primary idempotency key, while keeping `runDate` as grouping field. Scheduled runs use `runId=scheduled-{runDate}` (one per day); manual runs use `runId=manual-{runDate}-{uuid}` (multiple per day allowed). Schema changes: add `runId` and `triggerSource` to `GalleryImageRecord` and skip records. Asset paths include `runId` to prevent collision. Idempotency check changes from "does `runDate` exist?" to "does `runId` exist?". Multiple images per day share `runDate` but have distinct `runId` values. Frontend already handles multiple images per day (no code changes needed). Skip records still close the `runDate` (one skip per day, no multi-skip). Rate limits and API auth defined at API layer. Backward compatible: old images without `runId` default to `id`. Rollout sequence: (1) add `runId` field, (2) update idempotency check, (3) update bootstrap shell for scheduled runs, (4) deploy and verify with smoke proof, (5) build API endpoint with auth and rate limits.
- Multi-run idempotency contract design: Kyle designed backend contract for multi-run support using composite run identity. Run identification: each execution gets unique `runId` (format `{runDate}-{triggerSource}-{seq}` for readability). Trigger sources: `scheduled` (daily cron), `manual-api` (HTTP endpoint), `manual-cli` (admin override). Image identification: enhanced to `{runDate}-{imageName}` with collision suffix if needed. Skip record identity: `skip-{runId}`. Data shape: `GalleryImageRecord` gains `runId` and `triggerSource` fields (optional initially, required after Phase 3). Idempotency resolution: check `runId` exact match → no-op if found; new `runId` same `runDate` → proceed with generation; supports `allowMultiple=false` flag to reject multi-run. Retry behavior: same `runId` rerun = no-op; new `runId` same day = new generation; skip retryable via new `runId`. Git strategy: one commit per successful publish, multiple commits per day when multiple runs succeed. Backward compatible: Phase 1 (add `runId`, optional), Phase 2 (API route), Phase 3 (enforce uniqueness). Migration preferred format: `{runDate}-{source}-{seq}` for log readability; skip commits separate for audit trail; GitHub App extended for API permissions.
- API rollout lift sizing analysis: Kyle sized two implementation options for manual API trigger. Option 1 (Orchestrator-only, 2–3 days): add `runId` contract, update idempotency, validate multi-run, no infra changes. Option 2 (Full API integration, 1–2 weeks): Option 1 + Azure Function App, auth layer, ACA Job invocation, rate limiting, execution tracking. Key decisions: direct ACA Job invocation (not enqueue-first) because Job has built-in retry and queue adds little value; lock-free concurrency (orchestrator resolves via `runId` rather than advisory lock) because idempotency check is atomic and lock adds failure modes; API pre-checks `data/gallery.json` from GitHub before invocation to provide immediate feedback and prevent wasted job executions. Code touchpoints: `orchestrator/contracts.py` + `validation.py` + `main.py` for Option 1; new `api/trigger-run/*` + `infra/modules/apiFunction.bicep` for Option 2. Recommended sequence: prove Option 1 with manual CLI triggers first, add Option 2 behind feature flag once gates pass. Cost estimate: ~$0.04/month incremental (negligible). Hybrid approach recommended: start with Option 1, demonstrate multi-run safety, then build API layer incrementally.
